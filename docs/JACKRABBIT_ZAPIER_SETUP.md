# Jackrabbit-to-Django reporting setup

This guide connects the nine approved non-financial Jackrabbit activity signals
to the private Gyminators reporting dashboard. It is for the owner and the
technical administrator who manages production secrets and Zapier.

Jackrabbit remains the source of truth. This integration does not copy customer
profiles or payment records, and it does not calculate revenue.

## What this connection sends

The website accepts these event types:

| Jackrabbit activity | `event_type` | Required opaque identifiers |
| --- | --- | --- |
| Family created | `family_created` | `family_id` |
| Contact created | `contact_created` | `contact_id` |
| Student created | `student_created` | `student_id` |
| Lead created | `lead_created` | `family_id` |
| Student enrolled | `student_enrolled` | `student_id`, `class_id` |
| Student dropped | `student_dropped` | `student_id`, `class_id` |
| Student added to a waitlist | `waitlist_added` | `student_id`, `class_id` |
| Student removed from a waitlist | `waitlist_removed` | `student_id`, `class_id` |
| Student made inactive | `student_inactive` | `student_id` |

Optional opaque identifiers may be included when that trigger supplies them:
`family_id`, `contact_id`, `student_id`, `class_id`, and `enrollment_id`.
`location` may contain the business location associated with the event.

Never send any of the following:

- names, email addresses, phone numbers, street addresses, or birthdates;
- emergency contacts, attendance, skills, medical details, or notes;
- balances, charges, tuition transactions, payment methods, refunds, or revenue;
- Jackrabbit credentials, API keys, the inbound bearer token, or card data;
- an entire trigger object, `IncludeAllContacts` output, or any raw payload.

The endpoint rejects every field not named in its allowlist. That failure is a
privacy safeguard; do not work around it by placing private data inside an ID,
`location`, `source`, or `idempotency_key`.

## Before creating the Zaps

1. Deploy the current application, run migrations, and run `setup_roles`.
2. Confirm `JACKRABBIT_ORG_ID=154877` with the owner.
3. Generate a unique inbound secret of at least 32 characters. For example, on
   the production host:

   ```bash
   python3 -c 'import secrets; print(secrets.token_urlsafe(48))'
   ```

4. Put that value in `JACKRABBIT_WEBHOOK_TOKEN` in the production `.env`. Do not
   commit it, paste it into website content, or reuse a Jackrabbit password/API
   key. This token is generated by Gyminators specifically for Zapier to call
   this website.
5. Restart the application and verify configuration:

   ```bash
   docker compose up -d --build
   docker compose exec app python manage.py check_jackrabbit_reporting
   ```

6. In Django admin, add approved report viewers to **Reporting Managers**. Add
   **Website Managers** separately only if they also edit public content.
7. Record the owner's retention decision for live events and encrypted off-site
   backups. The application does not automatically purge reporting events. If
   indefinite retention is not approved, implement and test the approved purge/
   backup-expiration procedure before activating any Zap.

The production endpoint is:

```text
https://YOUR-DOMAIN/api/jackrabbit/v1/events/
```

Use HTTPS only. Do not send test events to the public Gyminators domain until
the owner has approved the connection and the production database is backed up.

## Request contract

Each Zap sends an HTTP `POST` with these headers:

```text
Authorization: Bearer YOUR_JACKRABBIT_WEBHOOK_TOKEN
Content-Type: application/json
```

The JSON object may contain only:

| Field | Rule |
| --- | --- |
| `schema_version` | Required integer; currently `1` |
| `event_type` | Required; one of the nine values in the table above |
| `idempotency_key` | Required stable, non-private string for this event occurrence |
| `occurred_at` | Required ISO-8601 date/time with timezone, such as `2026-07-13T14:25:00-04:00` |
| `source` | `trigger` for a live Zap or `backfill` for an approved historical load |
| `family_id` | Optional except where required above |
| `contact_id` | Optional except where required above |
| `student_id` | Optional except where required above |
| `class_id` | Optional except where required above |
| `enrollment_id` | Optional opaque enrollment identifier |
| `location` | Optional business location, never a home address |

Identifiers may be strings or integers. Do not prepend a name or email to make
an identifier easier to read. The endpoint stores only a one-way hash of the
idempotency key.

### Example creation event

```json
{
  "schema_version": 1,
  "event_type": "student_created",
  "idempotency_key": "student_created:482910",
  "occurred_at": "2026-07-13T14:25:00-04:00",
  "source": "trigger",
  "family_id": "240017",
  "student_id": "482910"
}
```

### Example enrollment event

```json
{
  "schema_version": 1,
  "event_type": "student_enrolled",
  "idempotency_key": "student_enrolled:7715001",
  "occurred_at": "2026-07-13T14:31:00-04:00",
  "source": "trigger",
  "family_id": "240017",
  "student_id": "482910",
  "class_id": "93011",
  "enrollment_id": "7715001",
  "location": "Main Gym"
}
```

The numbers above are fictitious. Never copy real customer identifiers into
documentation, tickets, or chat.

## Build the nine Zaps

Create one Zap for each row in the first table. Trigger labels and available
output fields can change in Zapier, so select the matching Jackrabbit activity
and verify its test output before mapping. Do not assume that similarly named
fields contain the same identifier.

For each Zap:

1. Choose Jackrabbit as the trigger app and connect the owner-approved
   organization using its normal Jackrabbit/Zapier authorization procedure.
2. Choose the matching activity trigger and test it with a controlled record.
3. Confirm the output contains the required opaque identifiers. If it does not,
   stop and do not activate that Zap.
4. Add **Webhooks by Zapier** as the action. Use a custom request if necessary so
   the body is sent as JSON rather than form fields.
5. Set method to `POST`, enter the endpoint above, add the Authorization and
   Content-Type headers, and map only the approved JSON fields.
6. Use the trigger's actual event/effective timestamp when available. It must
   include `Z` or a numeric UTC offset. If the trigger does not expose an event
   timestamp, use Zapier's run timestamp and record that limitation in the
   launch notes.
7. Send a test. A new event returns HTTP `201` with `{"status":"created"}`.
   Re-sending the exact event returns HTTP `200` with
   `{"status":"duplicate"}`.
8. Open `/dashboard/reporting/`, expand **View Zapier coverage by metric**, and
   confirm only the intended metric changed.
9. Name the Zap clearly, activate it, and record its owner and activation time
   without recording secrets or customer identifiers.

Activate one Zap at a time. This makes it possible to identify a bad mapping
before it affects multiple metrics.

### Stable idempotency keys

Zapier retries must not create multiple events. Build `idempotency_key` from
stable, opaque source fields—not a random value and not a person's information.
Recommended patterns are:

| Event | Preferred key pattern |
| --- | --- |
| Family/contact/student/lead created | event type plus its stable record ID |
| Enrolled/dropped | event type plus a stable enrollment or activity ID |
| Waitlist added/removed | event type plus a stable waitlist/activity ID |
| Student inactive | event type plus student ID and the effective status-change timestamp |

If an enrollment or waitlist activity ID is unavailable, combine event type,
student ID, class ID, and the source's effective event timestamp. Do not use a
Zap run timestamp alone: a later retry or replay could then look like a new
business event.

The server includes the configured organization and event type when hashing the
key, so the same opaque enrollment ID can safely be used for both an enrollment
and its later drop.

## Verify all nine feeds

After activation, a technical administrator runs:

```bash
docker compose exec app python manage.py check_jackrabbit_reporting
```

The command reports configuration, first/last event times and counts by event
type without printing source identifiers. In the browser, verify:

- only authorized users can open `/dashboard/reporting/`;
- each connected metric has a tracking start and recent delivery time;
- an unconnected metric says **Awaiting this Zap/backfill**, not zero;
- enrollments minus drops equals the displayed net enrollment activity;
- waitlist movement is labeled as activity, not current waitlist size;
- inactive/drop signals are not described as confirmed churn;
- class tuition is labeled as a published listing price, not revenue.

Create a short owner-approved test procedure for each trigger. Delete or correct
test business records in Jackrabbit according to Jackrabbit's procedures; do
not edit imported Django event rows to conceal a mapping problem.

## Historical backfill

Live triggers normally establish a forward-looking tracking start. A backfill
is optional and must be approved because it changes how users interpret the
dashboard.

- Use only Jackrabbit search/export capabilities available to the authorized
  organization and reviewed by the owner.
- Current Jackrabbit/Zapier searches may help find contacts, students, or
  enrollments, but a current-record search is not necessarily a complete history
  of creation, drop, waitlist, lead, or inactive events.
- Do not assume the family-created feed can be fully reconstructed from a broad
  family listing. If the source does not expose a trustworthy event and original
  timestamp, leave the metric's historical period uncovered.
- Do not invent occurrence dates or turn a current snapshot into historical
  activity. Record an honest coverage start instead.
- Set `source` to `backfill`, retain the same strict field allowlist, and use a
  deterministic idempotency key so a rerun is safe.
- Backfill in small batches, compare counts with the approved source report, and
  stop if the website returns validation errors.

After a backfill, document which event types and date ranges were covered. The
dashboard cannot infer missing history merely because it received one old row.

## Token rotation

Rotate the inbound secret after suspected exposure and on the owner's approved
schedule:

1. Generate a new random token of at least 32 characters.
2. Move the old current value to `JACKRABBIT_WEBHOOK_PREVIOUS_TOKEN` and put the
   new value in `JACKRABBIT_WEBHOOK_TOKEN`.
3. Restart the application. Both values are now accepted temporarily.
4. Update all nine Zaps to the new Authorization header and test each one.
5. Confirm deliveries on the coverage panel.
6. Clear `JACKRABBIT_WEBHOOK_PREVIOUS_TOKEN` and restart again.

Never leave the previous token configured indefinitely. If exposure is
suspected, disable the affected Zaps and remove the exposed token immediately;
do not use the overlap period.

## Class schedule synchronization

The class schedule, openings, waitlist/full state, and published tuition come
from Jackrabbit's public class feed. This is independent of the nine event Zaps
and does not use the inbound token.

Production's `class-sync` service refreshes the cache about every 15 minutes.
For a one-time refresh or status check:

```bash
docker compose exec app python manage.py sync_jackrabbit_classes
docker compose exec app python manage.py check_jackrabbit_reporting
```

A failed or malformed feed never retires the last successful class copy. The
first valid feed that omits a cached class leaves it visible with a warning; a
second consecutive valid omission confirms retirement. The dashboard marks old
or uncertain data; use Jackrabbit directly until the feed is healthy. Do not use
published tuition as collected income or openings as total capacity.

For a per-day class, Jackrabbit's main tuition fee is the one-day total and each
`day_N` value is the total charge for selecting N days. The website displays the
range of those totals and never labels them as per-day rates.

## Troubleshooting responses

| Response | Meaning and safe action |
| --- | --- |
| `201 created` | A new valid event was stored |
| `200 duplicate` | The same idempotency key was already accepted; no second row was created |
| `400` | The JSON, schema, timestamp, event type, required ID, source, or allowlist is invalid; fix the mapping rather than adding fields |
| `401` | The Bearer header is missing or does not match the configured current/previous token |
| `409` | The same idempotency key arrived with different event data; pause the Zap and correct its stable-key mapping |
| `413` | The payload is too large; this usually indicates an accidental raw object or extra data |
| `415` | The request was not sent as `application/json` |
| `503` | Reporting or the current inbound token is not configured on the website |

If deliveries stop, check the Zap task history, application status, token
configuration, and `check_jackrabbit_reporting`. Do not paste a failed payload
containing customer data into logs, email, chat, or a support ticket. Record the
outage interval and reconcile supported events before relying on totals again.

To suspend all reporting intentionally, set
`JACKRABBIT_REPORTING_ENABLED=false`, restart, and pause the nine Zaps. This
also idles automatic and manual class refresh. Existing events and the last
class copy are retained until an owner-approved retention process removes them.
